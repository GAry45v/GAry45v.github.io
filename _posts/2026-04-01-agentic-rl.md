Agentic RL 将强化学习从“优化单次回答”扩展到“优化完整任务轨迹”。Agent 需要理解目标、维护状态、选择工具、处理失败，并在正确的时机停止。最终结果可能只由少数关键决策决定，因此轨迹级信用分配成为核心问题。

本文只介绍通用架构与公开经验，不包含任何真实业务流程、内部工具定义、私有 Prompt、线上数据或训练参数。

## 1. Agent 学习的对象是一条轨迹

一条典型轨迹可以表示为：

$$
\tau = (s_0, a_0, o_1, s_1, a_1, \ldots, s_T)
$$

其中 $s_t$ 是当前上下文与环境状态，$a_t$ 是模型生成的文本或工具动作，$o_{t+1}$ 是环境返回的观察。训练信号既可以来自最终状态，也可以来自中间步骤。

与单轮问答相比，Agent 还需要学习：

- 是否应该调用工具；
- 应选择哪个工具以及如何构造参数；
- 工具失败后如何恢复；
- 信息不足时是否需要澄清；
- 何时继续、回退、重试或停止。

这里的状态不只是可见对话。它还可能包含工具执行后的环境状态、剩余预算、权限、未完成子目标和历史错误。模型未必能观察全部状态，因此很多 Agent 任务本质上是部分可观察的序列决策问题。

### 轨迹需要保存什么

为了让 rollout、训练和评估对同一条轨迹达成一致，建议使用结构化事件，而不是只保存拼接后的纯文本：

```json
{
  "task": {"instruction": "...", "initial_state": "..."},
  "events": [
    {"type": "assistant", "content": "..."},
    {"type": "tool_call", "tool": "...", "arguments": {}},
    {"type": "tool_result", "status": "ok", "content": "..."}
  ],
  "final_state": "...",
  "rewards": {"outcome": "...", "constraints": "..."}
}
```

公开示例只需保留抽象字段。真实轨迹还应区分模型可见内容、仅供 Verifier 使用的环境真值，以及不得进入上下文的敏感信息。

### Action mask 与训练边界

通常只对模型生成的 assistant 文本和工具调用计算策略损失，system、user、工具返回与环境事件只作为条件。若 action mask 错位，模型可能被训练去预测工具结果，或者关键调用没有梯度。

因此每条轨迹不仅要能“回放”，还要能逐 Token 解释哪些位置由策略生成、哪些位置参与奖励、哪些位置只是环境观察。

## 2. Harness 是模型能力的一部分

Agent 的表现不仅由模型参数决定，也由 Harness 决定。Harness 包含上下文构造、工具协议、状态管理、错误处理、权限控制和轨迹记录。

同一个模型在不同 Harness 下可能表现差异很大。训练与评估必须保持以下契约一致：

- 工具名称、参数 schema 和返回格式；
- system 指令与上下文组织方式；
- 超时、重试和错误语义；
- 环境状态的可见范围；
- 终止条件与安全边界。

如果契约在训练和部署之间发生变化，模型学到的策略很难稳定迁移。

### Context engineering

Agent 的上下文窗口是一种有限资源。Harness 需要决定哪些信息常驻、哪些按需检索、哪些压缩成摘要，以及工具结果保留到什么粒度。常见问题包括：

- 历史工具结果过长，挤掉最初任务与关键约束；
- 摘要丢失实体、数值或未完成子目标；
- 同一事实在不同消息中冲突，模型不知道以哪个为准；
- 错误堆栈和无关日志占据注意力；
- 系统指令与工具描述重复，增加行为歧义。

上下文策略必须版本化，并与训练模板共同评测。否则一次看似无害的 Prompt 重构就可能改变状态表示，使已有策略失效。

### 工具设计会塑造策略

工具应有单一明确职责、可判定的参数约束和稳定的错误语义。过于宽泛的工具让每次调用都像重新写程序；过于碎片化的工具又会拉长轨迹并放大一步错误。

工具返回应区分“调用失败”“执行成功但结果为空”和“任务条件不满足”。如果这些状态都返回相似自然语言，模型很难学会正确恢复策略。

## 3. 从种子任务到训练轨迹

Agentic RL 的数据通常来自任务定义、策略 rollout 和环境反馈。一个通用流程是：

1. 编写可验证的任务与初始状态；
2. 在隔离环境中运行策略；
3. 保存动作、观察、状态变化和错误；
4. 使用 Verifier 或 Rubric 标注轨迹；
5. 过滤不可复现、不可判定或存在环境故障的样本；
6. 将有效轨迹送入训练与回归评测。

公开数据应使用合成或授权环境，避免真实用户信息、内部接口和生产日志进入训练集或文章示例。

### Cold start 与在线改进

纯随机或能力不足的策略很难产生成功长轨迹，只有终局奖励时尤其如此。常见 cold-start 数据包括人工示范、强模型生成后验证的轨迹，以及由任务程序自动构造的最短成功路径。

这些轨迹可先用于 SFT，让模型掌握基本工具语法和任务分解；随后再通过 RL 学习错误恢复、路径选择与效率。SFT 负责建立可探索的初始策略，RL 负责在反馈下重新分配行为概率，两者解决的问题不同。

### 轨迹去重与污染控制

Agent 数据的重复不能只比较最终回答。相同动作序列、相同环境状态变化或相同失败前缀，都可能构成近似重复。若少数任务模板产生大量 rollout，模型会记住路径模式而非学习一般策略。

训练与评测还应按任务模板、环境实例和生成来源隔离，防止同一初始状态或等价目标跨集合泄漏。

## 4. Sandbox 与环境可复现性

Agent 会产生外部动作，因此训练环境必须隔离、可重置并可审计。理想的 Sandbox 具备：

- 确定的初始状态；
- 可控的工具响应与故障注入；
- 任务结束后的状态检查；
- 对危险或不可逆操作的拦截；
- 完整的动作和观察日志；
- 快速重置与并行执行能力。

如果环境本身不稳定，奖励会混入环境噪声，模型可能学到错误因果关系。

### 环境应当可判定，而不只是可运行

任务结束后，Sandbox 必须能回答“目标是否真的完成”。这通常需要独立于模型输出的状态检查器。模型声称操作成功并不构成成功，Verifier 应读取环境终态、执行测试或比较状态差异。

环境还需要区分策略失败与基础设施失败。超时、服务不可用和随机环境错误不应直接记成负奖励，否则模型会为自己无法控制的噪声承担信用。

### 可重放与故障注入

给定相同初始状态和动作序列，环境应尽可能产生一致结果。对不可完全确定的工具，应记录必要的响应快照或随机源。

在稳定回放的基础上，可以主动注入超时、空结果、权限拒绝和部分成功，评估模型是否能读取错误语义、调整计划并避免无限重试。

## 5. User Simulator：训练澄清与多轮协作

有些任务必须依赖后续用户信息，单一静态 Prompt 无法覆盖。User Simulator 可以根据隐藏目标、已知信息和对话状态生成回复，用于训练澄清、确认与纠错能力。

一个可靠的 Simulator 应明确：

- 用户拥有但尚未提供的信息；
- 哪些问题可以回答，哪些问题应拒绝或表示不知道；
- 用户偏好是否会在对话中变化；
- 何时确认任务完成，何时指出错误；
- 最大交互预算和停止条件。

Simulator 不能只追求语言自然。如果它会主动泄露答案、无条件配合 Agent，或在同一状态下给出矛盾回复，训练得到的是利用模拟器的策略，而不是与真实用户协作的能力。

评测时应更换 Simulator 指令、模型或行为风格，并保留部分人工交互集，检查策略是否过拟合某一种模拟用户。

## 6. Reward：结果、过程与约束

Agentic RL 的奖励通常由多个部分组成：

- **结果奖励**：目标状态是否达成；
- **过程奖励**：关键步骤是否正确；
- **约束奖励**：权限、安全和格式规则是否满足；
- **效率信号**：是否存在无效调用、循环或冗余步骤；
- **语义评价**：无法完全程序化的质量维度。

Rubric 可以把抽象质量拆成可检查的维度，再通过规则、模型或人工进行评估。

![Rubric-based data and reward workflow](/images/blogs/rl/rubric-pipeline.png)

这张图表示一种通用思路：从种子数据和 Rubric 出发构造评分器，通过离线筛选获得 RL 数据，再根据 rollout 中的新失败模式更新 Rubric。它不是任何特定生产系统的实现。

### Reward Model 的两种划分

按评分对象，可以分为：

- **Outcome Reward Model**：只判断最终状态或最终回答；
- **Process Reward Model**：判断中间步骤、计划或工具动作。

按输出形式，又可以分为：

- **Discriminative RM**：直接输出类别、偏好或标量分数；
- **Generative RM**：先生成依据、错误分析或 Rubric 逐项判断，再汇总结果。

Outcome RM 更接近最终目标，但反馈稀疏；Process RM 提供更密集的信用，却更容易把某种固定解题路径误当作唯一正确过程。Generative RM 可解释性更好，但解析、稳定性和推理成本也更高。

### Rubric 的生命周期

Rubric 不是一次写完的静态 Prompt。一个可维护的流程包括：

1. 从任务定义中提取必须满足的成功条件；
2. 用失败样例检验各条标准是否有区分度；
3. 删除彼此重叠或不可观察的条目；
4. 对规则、Verifier、Judge 和人工标注进行一致性检查；
5. rollout 出现新漏洞后，新增回归样例并更新 Rubric；
6. 版本化评分器，避免新旧奖励混在同一训练批次中。

若一条 Rubric 无法从轨迹或终态中可靠判断，就不应伪装成精确奖励。

## 7. Verifier 与 LLM-as-a-Judge

能程序化验证的内容应优先使用确定性 Verifier，例如：

- 最终环境状态；
- 单元测试或执行结果；
- JSON schema 与参数类型；
- 工具调用的前置条件；
- 文件或数据库状态变化。

LLM-as-a-Judge 适合评价相关性、完整性和表达质量，但需要注意位置偏差、长度偏差、自我偏好和提示敏感性。可靠做法包括使用明确 Rubric、打乱候选顺序、保留理由、抽样人工复核，并定期检查 Judge 与真实目标的相关性。

### 防止奖励被利用

模型会优化评分器实际测量的信号，而不是设计者心中的意图。常见 reward hacking 包括：

- 在最终回答中声称任务完成，但环境状态没有变化；
- 重复 Rubric 关键词以迎合语义评分；
- 用更长解释换取完整性分数；
- 避开困难步骤，转而触发容易通过的检查；
- 利用工具或 Verifier 的边界输入获得异常结果。

对抗方法包括隐藏部分测试、使用独立终态验证、对评分理由做一致性检查、比较多个评分器，以及定期人工查看高奖励但低可信的轨迹。

Verifier 本身也需要测试。可以为每条规则准备应通过、应失败和边界样例，并确保环境更新后回归集仍然有效。

## 8. 长轨迹中的信用分配

最终失败不代表每一步都错，最终成功也不代表过程安全。信用分配可以结合：

- 任务终局奖励；
- 子目标完成信号；
- 工具执行结果；
- 关键决策点的局部验证；
- 反事实重放或失败前缀分析。

过程奖励越细，越容易提供学习信号，也越容易被模型利用。设计时需要确认每个中间目标确实服务于最终目标。

### 终局奖励如何回传

最简单的方法是把终局优势广播给整条模型动作序列。这样无需判断哪一步关键，但无效措辞和正确决策会得到相同方向的更新。

更细的方法是以工具调用、子目标或状态变化为边界，为不同片段分配奖励。它要求环境能可靠标记“这一步带来了什么变化”，否则细粒度奖励只是更复杂的噪声。

### 失败前缀与反事实

对失败轨迹，可以寻找第一个不可恢复的决策点：在此之前轨迹仍可能通过另一动作成功，在此之后则已经偏离目标。通过从中间状态重放多个候选动作，可以估计哪些步骤真正改变了成功概率。

这类反事实分析成本较高，更适合用来生成高价值训练样本、训练步骤级 Verifier，或解释典型失败，而不是对所有轨迹无差别执行。

## 9. 课程学习与失败驱动采样

早期策略适合从短轨迹、明确工具和稳定反馈开始。能力提升后，再逐步加入多工具依赖、部分可观察状态、可恢复错误和长程任务。

更有效的课程往往由失败类型驱动：统计模型在哪些状态下误判、循环、越权或过早停止，再补充能够区分这些行为的任务。难度应随模型变化动态更新，而不是固定一次后长期不变。

可将失败粗分为：任务理解错误、计划错误、工具选择错误、参数错误、观察理解错误、恢复失败、违反约束和终止判断错误。只有把失败落到可行动类别，数据补充和奖励调整才有明确方向。

采样时既要提高高价值失败的覆盖，也要保留已经掌握的基础任务。否则训练分布会追着最新错误移动，导致旧能力反复遗忘。

## 10. Agent 评估不能只看最终成功率

完整评估至少应覆盖：

- 任务成功率与状态正确性；
- 工具选择和参数正确性；
- 平均步骤数与无效调用率；
- 错误恢复与澄清能力；
- 权限、安全和不可逆操作；
- 不同任务长度与难度分层；
- 环境扰动和工具故障下的鲁棒性；
- 轨迹可解释性与可复现性。

最终成功率是必要指标，但不足以说明 Agent 的行为是否可靠。

### 分层评估

可以把评估拆成四层：

1. **组件层**：工具 schema、状态检查器和 Reward Model 是否正确；
2. **步骤层**：给定状态时，模型能否选择合理的下一动作；
3. **轨迹层**：完整任务能否成功，路径是否安全高效；
4. **系统层**：在并发、故障、上下文压力和分布变化下是否稳定。

步骤层评测便于定位问题，轨迹层评测最接近真实能力，系统层评测则暴露 Harness 与环境瓶颈。只做其中一层，很容易把组件缺陷误判为模型能力问题。

### 评估中的污染与随机性

Agent 任务可能因搜索结果、外部服务或 Simulator 随机性而变化。每次评测应记录环境版本、工具版本、随机源和模型版本，并对关键任务进行多次运行。

结果报告不应只给均值，还应展示任务级成功率、置信区间、错误类型和路径长度分布。这样才能区分稳定提升与少数随机成功。

## 11. 从失败到数据飞轮

一个可持续的 Agentic RL 闭环通常是：

```text
evaluation failures
  -> failure attribution
  -> task and rubric updates
  -> sandboxed rollout
  -> verifier filtering
  -> SFT / RL data mixture
  -> regression evaluation
```

并非每个失败都应该直接变成 RL 样本：

- 知识缺失可能更适合检索或数据补充；
- 工具 schema 不清晰应先修 Harness；
- Verifier 误判应先修评分器；
- 环境不稳定应先修基础设施；
- 策略在多个有效动作间选择不佳，才更适合通过偏好或 RL 调整。

把所有问题都归因于模型并继续训练，会让系统复杂度上升，却不一定提升真实能力。

## 12. 总结

Agentic RL 的本质，是让模型在环境中学习可验证的决策策略。模型、Harness、环境、Verifier 和数据分布共同构成训练系统，任何一环变化都可能改变最终行为。

优秀的 Agentic RL 实践不只追求更高奖励，还要保证轨迹可解释、环境可复现、动作可审计、失败可归因。只有这样，训练得到的能力才有机会从实验环境迁移到真实任务。
