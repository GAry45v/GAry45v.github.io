大语言模型强化学习的目标，是让模型根据可验证反馈调整生成策略。与 SFT 不同，RL 不要求每个输入都有唯一标准答案，而是通过奖励比较不同输出的质量。

本文聚焦公开算法与通用工程原则，不包含任何真实业务数据、内部指标或训练超参数。

## 1. 从策略梯度开始

将模型视为策略 $\pi_\theta$：给定状态或上下文，模型逐步生成 Token 或动作。训练希望提高高奖励轨迹的概率，并降低低奖励轨迹的概率。

对一条生成序列 $y=(y_1,\ldots,y_T)$，它在策略下的概率可以分解为：

$$
\pi_\theta(y\mid x)=\prod_{t=1}^{T}\pi_\theta(y_t\mid x,y_{<t})
$$

策略梯度的核心形式是：

$$
\nabla_\theta J(\theta)
=\mathbb{E}\left[\sum_t \hat A_t\nabla_\theta
\log \pi_\theta(y_t\mid x,y_{<t})\right]
$$

直觉很简单：优势为正的动作被提高概率，优势为负的动作被降低概率。真正困难的是如何得到低方差、低偏差且不容易被奖励漏洞污染的 $\hat A_t$。

优势函数刻画某个动作相对于当前基线的好坏：

![Advantage function](/images/blogs/rl/advantage-function.png)

基线能够降低梯度估计的方差。对于长文本或工具轨迹，优势估计还承担信用分配的作用：成功或失败究竟应归因于哪些决策。

在只有终局奖励的场景中，同一条序列的 Token 往往共享相近的轨迹级优势。这种做法简单，但会把关键推理步骤、无关措辞和格式 Token 一起强化。若任务能够提供中间验证信号，才有条件进一步做更细粒度的信用分配。

## 2. On-policy 与 Off-policy

**On-policy** 方法使用当前策略生成的数据进行更新。它减少了分布偏移，但采样成本较高，旧数据也会很快失效。

**Off-policy** 方法可以复用历史数据，但必须处理行为策略与当前策略之间的差异。重要性采样能够进行校正，但权重过大时会显著增加方差。

行为策略 $\pi_{\text{old}}$ 与当前策略 $\pi_\theta$ 的差异通常通过概率比刻画：

$$
r_t(\theta)=
\frac{\pi_\theta(y_t\mid x,y_{<t})}
{\pi_{\text{old}}(y_t\mid x,y_{<t})}
$$

当概率比远离 $1$ 时，说明新策略已经偏离生成这批数据的策略。裁剪目标并不能消除分布偏移，但能限制单批旧轨迹产生过大的更新。

在 LLM 场景中，实践常采用折中方案：限制数据的新旧程度，记录生成策略版本，并通过裁剪或过滤控制策略偏移。

这要求 rollout 数据保存生成时的 Token、旧策略 log probability、mask、奖励分量和策略版本。如果重新分词、聊天模板变化或 log probability 与实际采样序列错位，训练仍可能运行，却已经不再优化预期目标。

## 3. 为什么使用 GRPO

Group Relative Policy Optimization（GRPO）对同一输入采样一组输出，并使用组内相对奖励构造优势。它避免单独训练价值模型，适合可并行验证的推理任务。

![GRPO objective](/images/blogs/rl/grpo-objective.png)

一个抽象的训练循环包括：

1. 从任务分布中采样输入；
2. 使用当前策略生成多条候选响应；
3. 通过规则、执行器或 Judge 计算奖励；
4. 在组内标准化奖励，得到相对优势；
5. 使用裁剪目标更新策略；
6. 监控策略漂移、熵、长度和奖励分布。

公式中的裁剪范围、KL 系数和采样配置都依赖具体任务。公开分享时，更有价值的是说明选择依据与稳定性观察，而不是给出脱离上下文的固定数值。

### 组内相对优势

设同一输入采样得到 $G$ 个回答，奖励为 $R_1,\ldots,R_G$。一种常见的组内标准化形式是：

$$
\hat A_i=
\frac{R_i-\operatorname{mean}(R_1,\ldots,R_G)}
{\operatorname{std}(R_1,\ldots,R_G)+\epsilon}
$$

它把“这个回答在全局有多好”转化为“它相对同组候选有多好”。这样可以降低不同问题之间奖励尺度不一致的影响，但也带来两个边界问题：

- 同组奖励完全相同时，样本几乎没有相对学习信号；
- 如果组内只有一个偶然高分样本，标准化可能放大噪声或奖励漏洞。

因此，组内成功率、奖励方差和有效优势比例比单纯的平均奖励更值得监控。

### Token 级与序列级更新

经典 PPO 风格目标通常在 Token 级计算概率比，再对有效 Token 聚合。它能表达细粒度变化，但长序列会贡献更多项，且某个局部概率比可能主导整条轨迹。

序列级方法则先聚合整段回答的 log probability 或概率比，再进行裁剪和归一化。它更贴近“一个回答对应一个终局奖励”的设定，也能减轻长度带来的权重失衡，但会牺牲局部信用分配能力。

两种粒度没有绝对优劣。选择时应明确奖励究竟对应 Token、步骤还是完整回答，并检查长度变化是否悄悄改变了优化权重。

### KL 约束与参考策略

KL 约束用于限制策略远离参考模型，减少语言能力退化和异常模式放大。它可以进入训练目标，也可以只作为监控与早停信号。

需要区分三种“距离”：当前策略与 rollout 策略的距离、当前策略与参考策略的距离，以及最终行为与 SFT 基线的差异。三者反映的问题不同，不能用一个 KL 数字替代全部诊断。

## 4. 奖励决定模型会学成什么

奖励函数不是单纯的评测器，它定义了优化方向。常见信号包括：

- **Outcome reward**：最终答案或环境状态是否正确；
- **Process reward**：中间推理或工具步骤是否合理；
- **Format reward**：输出是否满足结构化约束；
- **Verifier reward**：程序、测试或规则检查是否通过；
- **LLM-as-a-Judge**：难以程序化的质量维度；
- **Cost penalty**：无效步骤、重复调用或过长轨迹的成本。

单一奖励很容易被模型钻空子。组合奖励时，应持续检查各分量的相关性、尺度和可被利用的漏洞。

一种更清晰的组织方式，是把奖励拆成三层：

1. **任务目标**：答案正确、环境目标达成或测试通过；
2. **行为约束**：格式、安全、工具协议与必要步骤；
3. **优化偏好**：简洁性、成本、可读性与风格。

任务目标应拥有最高解释优先级。若模型为了获得格式分或 Judge 偏好而牺牲正确性，即使总奖励提高，也不应被视为有效进步。

### Rule、Verifier 与 Judge 如何组合

确定性规则适合判断结构和硬约束，执行型 Verifier 适合判断可运行结果，LLM-as-a-Judge 适合判断语义质量。三者可以形成级联：先用低成本规则拒绝明显无效输出，再运行环境验证，最后只对无法程序化的维度调用 Judge。

奖励融合前应先单独验证每个信号。建议保留每个分量、评分理由和失败类型，而不是只保存加权总分，否则训练异常时很难判断是哪一个评分器发生了漂移。

## 5. 常见不稳定来源

### 奖励稀疏

如果只有最终成功才有奖励，长轨迹中的有效动作难以获得信用。可以引入可验证的阶段性目标，但要避免过程奖励替代真实目标。

### 奖励尺度漂移

不同任务或不同 Judge 的分数分布可能不一致。应监控分布变化，并避免少数极端值主导更新。

组内标准化只能消除部分尺度问题。如果某一任务的奖励几乎恒定，它对梯度贡献可能很小；如果另一任务的奖励方差很大，它会持续主导有效样本。需要按任务切片检查优势分布，而不是只看全局直方图。

### 长度偏置

Judge 可能偏好更长、更完整的回答，模型随后会通过增加长度获得奖励。评估应同时观察正确性、简洁性和无效 Token 比例。

### 策略坍塌

模型可能快速收敛到少数高频模板，表现为熵下降、多样性消失或拒答增加。此时需要检查数据分布、奖励漏洞和更新幅度，而不是只继续训练。

### 正负优势失衡

如果大部分候选都成功或都失败，组内优势会退化；如果正优势样本只来自少数模板，模型又可能迅速强化单一模式。应跟踪正、负和近零优势的 Token 占比，以及它们来自哪些任务和回答长度。

### 可验证奖励与真实质量错位

程序验证通常可靠，但只覆盖可观测部分。例如测试通过不一定代表解释正确，格式合法也不代表工具选择合理。训练集中的 Verifier 覆盖范围必须与最终能力目标区分开，否则模型会把“通过检查”当成任务本身。

## 6. 样本筛选与课程学习

不是所有样本都同样适合 RL。过易样本缺乏学习信号，过难样本可能只产生噪声。可以根据当前策略的成功率、奖励方差或验证器反馈，将样本划分为不同难度，并动态调整采样。

课程学习的目标不是永久按“由易到难”排序，而是维持有信息量的训练分布。随着策略能力变化，样本难度也应重新估计。

### 什么是有效 RL 样本

对组相对优化而言，最有信息量的往往是同一输入下同时出现成功与失败的组。全对样本和全错样本仍可用于评估、数据生成或后续难度调整，但未必能提供稳定的相对优势。

筛选时不能只保留高方差样本，否则训练分布会越来越偏向模型当前的边界区域。较稳妥的做法是同时保留能力覆盖、难度覆盖和失败覆盖，并定期用固定基准检查是否遗忘基础能力。

## 7. Rollout 系统与训练闭环

LLM RL 通常包含两个不同节奏的系统：负责大规模生成的 rollout 侧，以及负责梯度更新的 training 侧。两者之间至少需要传递：

```text
prompt and template version
generated token ids and masks
old-policy log probabilities
reward components and verifier traces
sampling metadata and policy version
```

训练侧更新权重后，rollout 侧需要按明确策略刷新模型。刷新过慢会增加 off-policy 偏移，刷新过频又可能造成系统空转。工程上应把策略版本、数据生成时间和消费位置记录到每个批次，使异常可以追溯到具体模型快照。

### 先验证数据对齐，再调算法

许多看似“RL 不稳定”的问题其实是数据对齐错误，例如：

- prompt 与 response mask 边界错误；
- reward 绑定到了同组的另一条回答；
- 旧策略 log probability 使用了不同模板重新计算；
- padding Token 参与了损失或长度归一化；
- rollout 与训练 Tokenizer 版本不一致。

建议构造极小的可解释批次，逐 Token 对比采样序列、log probability、优势、mask 和最终 loss。这个检查通常比直接扩大训练规模更能节省时间。

## 8. 评估训练是否真的有效

除总奖励外，还应观察：

- 独立测试集上的真实成功率；
- 奖励分量与人工判断的一致性；
- 输出长度、熵和重复率；
- 不同难度、领域和错误类型的表现；
- 对奖励扰动和 Judge 替换的鲁棒性；
- SFT 基线与 RL 模型之间的能力回归。

最重要的检查是：模型是否提升了目标能力，而不是只提升了奖励模型喜欢的特征。

评估应分为三层：训练监控用于发现数值异常，离线基准用于比较能力，隔离环境中的端到端评测用于验证真实行为。三层指标可能互相矛盾，这恰恰是需要定位的问题，而不是选择最好看的一个汇报。

## 9. 建议持续记录的诊断面板

- policy loss、梯度范数和更新后的概率比；
- 当前策略相对 rollout 策略与参考策略的 KL；
- Token 熵、回答长度和截断比例；
- 总奖励、各奖励分量及其相关性；
- 组内奖励方差、全对组、全错组和混合组比例；
- 正负优势比例及其在任务、长度和领域上的分布；
- Verifier 失败、Judge 失败和环境失败的独立计数；
- 固定回归集上的 SFT 能力保持情况。

这些指标的价值不在于越多越好，而在于能把“数据问题、奖励问题、策略更新问题和评估问题”区分开。

## 10. 总结

LLM 强化学习的难点通常不在公式，而在反馈质量、数据分布和稳定性控制。算法提供更新规则，Verifier 和评估体系决定这些更新是否有意义。

一个可靠的 RL 项目应能够解释：奖励从哪里来、哪些样本产生了学习信号、策略发生了什么变化，以及这种变化能否在独立环境中复现。
