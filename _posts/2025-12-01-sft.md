Supervised Fine-Tuning（SFT）常被理解为“准备指令数据，然后训练模型”。真正决定效果的，通常不是训练命令本身，而是任务边界、数据质量、格式一致性和评估闭环。

这篇文章只讨论可迁移的公开方法，不涉及任何公司的数据、业务规则、内部系统或训练配置。

## 1. 先定义模型应该学会什么

在收集数据之前，需要把目标能力写成可观察的行为。例如：

- 能否理解用户意图并给出相关回答；
- 能否遵循输出格式与安全约束；
- 能否在信息不足时主动澄清；
- 能否正确选择工具，并生成合法参数；
- 能否在任务结束后停止，而不是继续产生无效动作。

目标如果无法被检查，就很难通过 SFT 稳定地教给模型。

## 2. 数据来源与风险

常见数据来源包括公开数据、人工编写数据和合成数据。三者各有价值：

- **公开数据**适合建立通用能力，但需要检查许可、质量和分布差异；
- **人工数据**可控性高，适合覆盖关键行为和边界情况；
- **合成数据**扩展速度快，但容易复制生成模型的偏差与表达模式。

不论来源如何，发布或共享前都应移除个人信息、公司信息、内部接口、真实日志、私有 Prompt 和业务指标。真实交互数据尤其需要先做授权、脱敏和最小化处理。

## 3. 质量优先于数量

高质量样本至少应满足以下条件：

1. 指令明确，输入信息足够完成任务；
2. 回答正确，并与目标行为一致；
3. 不包含互相矛盾的规则；
4. 格式能够被训练与推理代码稳定解析；
5. 样本之间具有足够多样性，而不是模板的机械重复。

常见清洗步骤包括精确去重、语义去重、格式校验、长度异常检测、安全检查和人工抽检。模型评分可以辅助筛选，但不应成为唯一标准，因为 Judge 自身也会产生系统性偏差。

## 4. 数据后处理：从原料到训练样本

原始数据通常不能直接进入训练。一个可复用的处理管线可以拆成以下步骤：

```text
raw records
  -> schema validation
  -> privacy and safety filtering
  -> exact / semantic deduplication
  -> quality scoring
  -> format normalization
  -> train / validation split
  -> immutable dataset snapshot
```

### 精确去重与语义去重

精确去重适合发现完全相同的样本，可基于规范化文本或内容哈希实现。语义去重用于识别“表达不同但含义几乎相同”的样本，避免模型被少量模板主导。

语义去重不能只依赖单一相似度阈值。更稳妥的做法是先按任务、语言和长度分桶，再在桶内比较，并抽样检查被删除的样本，防止误删有效的困难样本。

### 质量过滤

质量过滤可以组合三类信号：

- 规则信号：结构、长度、字段完整性和安全检查；
- 模型信号：相关性、正确性、完整性和可读性评分；
- 人工信号：对关键任务和低置信样本进行复核。

Judge 的分数更适合用于排序和发现异常，不适合作为绝对真值。对于高风险数据，应保留评分理由与复核记录。

### 数据版本化

每次训练应绑定不可变的数据快照，并记录数据来源、清洗规则、模板版本和切分方式。否则模型变化无法归因，错误样本也难以追溯。

## 5. 数据质检：检查分布而不只是单条样本

单条样本正确，不代表数据集整体健康。数据级质检应覆盖：

- 输入、输出与 Token 长度分布；
- 任务类别与语言分布；
- 重复率和模板集中度；
- 格式错误、安全问题和拒答比例；
- 来源分布与潜在污染；
- 训练集和测试集之间的近似重复。

建议为每个数据版本生成固定格式的质量报告。报告不只给出平均值，还应展示分位数、长尾类别和异常样本，以便发现被总体指标掩盖的问题。

## 6. 格式一致性是隐藏的关键变量

聊天模型通常同时处理 system、user、assistant 和 tool 等角色。训练模板与推理模板必须一致，否则模型可能在训练集上表现正常，部署后却出现角色错位、特殊 Token 泄漏或工具调用格式错误。

建议在训练前自动检查：

- 每条对话的角色顺序是否合法；
- 特殊标记是否成对出现；
- 工具参数是否为有效的结构化数据；
- assistant 之外的 Token 是否按预期参与损失；
- 截断是否破坏了答案或工具调用闭环。

### Loss mask

对于对话式 SFT，通常只希望模型学习 assistant 的输出，而不把 system、user 或工具返回当作预测目标。Loss mask 必须与聊天模板共同验证，尤其要检查：

- assistant 起止位置是否正确；
- 多轮对话中每个 assistant 段是否被覆盖；
- 特殊 Token 是否被错误计入损失；
- 截断后是否留下不完整的监督区间；
- 工具调用和最终回答是否采用一致的监督策略。

一个常见的隐蔽错误是模板看起来正确，但 mask 偏移了一个 Token。训练损失仍会下降，模型行为却会逐渐异常。因此应在训练前把 token、role 和 mask 可视化抽检。

## 7. 数据混合与能力保持

只训练单一任务数据，容易让模型过度适应该任务，并损害原有的通用能力。数据混合应围绕能力覆盖来设计，而不是简单拼接文件。

一个实用思路是将数据划分为：

- 核心任务数据；
- 通用指令遵循数据；
- 安全与拒答数据；
- 边界情况和失败修复数据；
- 工具调用或结构化输出数据。

不同类型数据的比例没有通用答案，应通过独立验证集和能力回归测试决定。

混合数据时可以为每类数据维护独立的验证切片。这样能看出某次调整究竟提升了核心任务，还是只提高了占比最大的那类样本。

## 8. 训练阶段关注什么

具体学习率、批大小、训练轮数和并行策略依赖模型、硬件与数据分布，不适合脱离上下文给出固定数值。更重要的是观察：

- 训练损失与验证损失是否持续分离；
- 核心能力是否提升，同时通用能力是否退化；
- 长回答是否开始模板化或重复；
- 格式错误和拒答行为是否发生结构性变化；
- 不同数据子集的学习速度是否失衡。

保存多个检查点并进行统一评测，通常比只选择最终检查点更可靠。

### 梯度与有效 Token

除了损失，还应观察梯度范数、有效监督 Token 数、不同长度样本的贡献，以及各数据源的采样频率。大量 padding 或没有监督区域的样本，会让表面 batch size 与真实学习信号不一致。

### 过拟合不只表现为验证损失上升

生成模型的过拟合还可能表现为固定开场、答案长度趋同、过度拒答、格式僵化和复述训练模板。因此需要用行为评估配合损失曲线判断。

## 9. 并行训练、检查点与导出

当模型无法在单设备上训练时，常见并行方式包括数据并行、张量并行、流水线并行和参数分片。选择方案时应考虑模型结构、通信开销、序列长度与检查点格式，而不是只追求理论显存上限。

检查点需要保存足够的恢复信息，包括模型权重、优化器状态、调度器状态、随机数状态和数据迭代位置。恢复训练后应验证损失是否连续，并确认数据没有重复或跳过。

训练格式与部署格式可能不同。导出阶段要检查权重合并、Tokenizer、聊天模板、特殊 Token、精度转换和推理引擎兼容性。一个模型只有在目标推理栈中通过回归测试，才算真正完成训练。

## 10. 评估要覆盖输出与行为

单一平均分很容易掩盖问题。一个更完整的评估集合可以包含：

- 任务正确性；
- 指令遵循与格式合法性；
- 安全性与隐私保护；
- 工具选择和参数正确性；
- 长尾样本与对抗样本；
- 通用能力回归测试；
- 人工偏好与错误类型分析。

自动指标适合高频回归，人工评估适合发现指标没有覆盖的新问题。两者结合，才能形成可持续的数据迭代闭环。

## 11. 常见失败模式

- **训练集与评测集污染**：指标异常提高，但新任务没有收益；
- **模板不一致**：离线输出正常，部署后角色或工具格式错位；
- **数据源失衡**：模型被高频模板支配，长尾能力下降；
- **Judge 过拟合**：模型学会迎合评分器，而非提升真实质量；
- **截断策略错误**：答案尾部、工具结果或关键条件长期缺失；
- **只看最终检查点**：错过更早出现的最佳能力状态；
- **缺少回归集**：核心任务提高的同时，通用能力悄然退化。

这些问题通常无法靠增加训练步数解决，必须回到数据、模板和评估链路定位原因。

## 12. 可复用的经验

SFT 的核心不是“让模型记住更多答案”，而是用一致、可验证的数据定义行为。优秀的 SFT 流程往往具备三个特征：数据来源可追踪、训练格式可验证、效果变化可归因。

当模型表现不佳时，先检查任务定义、数据和评估，再考虑扩大训练规模。很多看似需要更多算力的问题，实际来自样本冲突、格式错位或评估盲区。
